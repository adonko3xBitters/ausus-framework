# AUSUS v0.2.0-rc.1 — Release candidate

Coupe release-candidate de [v0.2.0-beta.1](v0.2.0-beta.1.md) avec
**zéro changement runtime**. Toutes les classes PHP, le bundle
renderer React, le format de fil ViewSchema et la surface HTTP API
sont bit-à-bit identiques à beta.1. RC.1 a été promue après que
l'audit de readiness beta.1 a fermé toutes les erreurs factuelles
de documentation et les lacunes de backfill release-engineering.

## Ce qui a changé entre beta.1 et rc.1

Deux commits de cleanup sur `main` entre les deux tags :

1. `docs(rc): resolve all RC.1 audit findings` — corrigeait 3
   bloqueurs RC (label compat-matrix README, lien doc cassé,
   référence morte à `scripts/release-replay.sh`) + 5 non-bloqueurs
   (comptes d'étapes PR template, biais beta des snippets d'install,
   docstrings d'en-tête de scripts).
2. `chore(release): backfill beta.1 release artifacts` — généré
   `artifacts/releases/v0.2.0-beta.1.json` (manifest immuable) +
   `artifacts/releases/v0.2.0-beta.1/` (archive evidence 8 fichiers)
   pour que la chaîne de publication beta.1 soit auditable end-to-end.

Les deux commits sont documentation / artefacts uniquement. Aucun
fichier source de package, aucun workflow, aucun comportement CI,
aucun champ wire n'a changé.

description complète de pagination, filtrage, tri, ViewSchema

## Ce que "release candidate" signifie ici

RC.1 est la candidate vers la ligne stable pour `v0.2.0`. La
promotion de beta à rc porte un contrat user-facing :

> Aucune nouvelle API publique ne sera ajoutée, supprimée ou
> renommée entre `v0.2.0-rc.1` et `v0.2.0`. Bug fixes uniquement.
> ViewSchema reste à `1.2.0` ; le `peerSchemaVersion` du renderer
> reste à `^1.0.0`.

Si un défaut runtime est découvert en rc.1, le fix sort en `rc.2`
en suivant la même gate de readiness-audit. Si aucun défaut n'est
trouvé, une release stable future `v0.2.0` sera coupée depuis un
snapshot équivalent à rc.1 (modulo le tidying documentation pour
la ligne stable).

## Installation

La commande d'installation canonique reste inchangée depuis beta —
le flag `@beta` de Composer accepte les release candidates :

```bash
composer create-project "ausus/starter:^0.2@beta" myapp --stability=beta
cd myapp
composer boot
composer serve   # → http://localhost:8000
```

Le canal `@rc` est équivalent pour la ligne rc :

```bash
composer create-project "ausus/starter:^0.2@rc" myapp --stability=rc
```

Les deux résolvent `v0.2.0-rc.1`. Les projets beta existants
récupèrent rc.1 automatiquement au `composer update` (rc > beta en
stability Composer).

## Renderer

`@ausus/renderer-react@0.2.0-rc.1` est la release npm appairée avec
ce backend. Aucun changement TypeScript / build / peer-dependency.

## Limitations connues

Inchangées depuis [beta.1](v0.2.0-beta.1.md#limitations-connues) —
pagination par curseur, arbres de filtre booléens imbriqués, tri
par expressions calculées, et drivers MySQL / PostgreSQL restent
tous du travail post-stable.
